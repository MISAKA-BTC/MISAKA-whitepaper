## 概要

Misaka Network は、以下の3つの原則を軸に設計された Layer-1 ブロックチェーンです。

- **デフォルトでプライバシー** — すべてのトランザクションは、ポスト量子格子暗号プリミティブを用いて送信者・受信者・金額を秘匿します
- **軽量な参加** — バリデータノードおよびライトノードは完全なトランザクション履歴を保持せずに動作します。完全な履歴はオペレーター管理のアーカイブノードのみが保持します
- **量子耐性** — スタック全体で楕円曲線暗号を一切使用しません。Falcon-512・Kyber-768・SHA3-256 を全面採用しています

| パラメータ | 値 |
|-----------|-----|
| ブロック時間 | 180秒（3分） |
| ブロックサイズ目標 | 4 MB |
| トランザクションサイズ | 50〜200 KB |
| 目標スループット | 2〜10 TPS |
| コンセンサス | Tendermint-style BFT、バリデータ 4〜16台 |
| 署名 | Falcon-512（NIST PQC標準） |
| リング署名 | LaRRS（Module-LWEベース） |
| 鍵交換 | Kyber-768（CRYSTALS-Kyber） |
| ハッシュ | SHA3-256 |
| セキュリティレベル | ポスト量子128ビット |

## ガバナンスシステム

Misaka は **Cardano 流の「三権分立」モデル** を採用し、その三権構造を Misaka 独自のノード役割に対応させています。すべてのガバナンス投票は Falcon-512 署名を使用するため、ガバナンス層自体も量子耐性を持ちます。

### 3.1 参加者と役割

| 役割 | Misaka での対応 | ガバナンス上の権限 |
|------|----------------|-----------------|
| **提案者 (Proposer)** | デポジットをロックした任意のネットワーク参加者 | 新規バリデータの推薦、パラメータ変更、憲法改正の提案 |
| **バリデータ（SPO 相当）** | 現役バリデータノード | 技術的な実行可能性を審査し、⌊2N/3⌋ + 1 以上の多数決で承認 |
| **アーカイブガバナンス（CC 相当）** | 財団・初期運営ノード | ネットワーク憲法（基本方針）への準拠を監視し、違反提案に拒否権を行使 |

> 3つの役割は意図的に独立しています。同一提案に対して、バリデータとアーカイブガバナンスが兼任することは禁止されており、結託を防止します。

---

### 3.2 提案ライフサイクル

このライフサイクルは Cardano の「ガバナンス・アクション」フローをベースとし、Misaka のエポックベースのブロック時間に合わせて設計されています。

#### ① 提案の提出

アーカイブノードを含むネットワークノードを運用している者であれば誰でも、`GovernanceAction` トランザクションをブロードキャストすることで提案を提出できます。

```
GovernanceAction TX
├── action_type       : u8      // 例: 0x01 = バリデータ追加
├── deposit           : u64     // ロックする MSK トークン量（スパム防止）
├── anchor_url        : Vec<u8> // オフチェーン提案理由文書の URL
└── anchor_hash       : [u8;32] // 文書内容の SHA3-256 ハッシュ
```

- **デポジット** は提案が批准されれば返却され、投票期間内に否決された場合は没収されます。
- **アンカー** は、なぜそのノードを推薦するのか等の提案背景をオンチェーンで裏付けるため、すべての参加者から検証可能です。

#### ② 投票期間（エポックベース）

投票ウィンドウは **エポック**（デフォルト: 100ブロック ≈ 5時間）に揃えられます。投票期間中に即時オンチェーン適用は行われません。

現役バリデータは自身の Falcon-512 コンセンサス鍵を用いて投票します。

```rust
enum Vote { Yes, No, Abstain }

struct GovernanceVote {
    proposal_id  : [u8; 32],
    validator_id : [u8; 32],   // Falcon-512 公開鍵ハッシュ
    vote         : Vote,
    weight       : u64,        // 投票スナップショット時点のステーク量
    signature    : [u8; 1281], // Falcon-512 署名
}
```

#### ③ 批准：⌊2N/3⌋ + 1 閾値

エポック境界で、以下の **両方** が満たされた場合に批准が成立します。

1. **バリデータクォーラム**: ステーク加重 YES 票 ≥ バリデータ総ステークの 2/3
2. **アーカイブガバナンスの承認**: アーカイブガバナンスノードの過半数が憲法違反の異議を申し立てていないこと

この二重チェックにより、バリデータの超多数決でもネットワーク憲法の根本方針を侵害するような変更を強行できなくなります（Cardano の Constitutional Committee による拒否権に相当）。

#### ④ 発効

批准された提案は **次のエポック境界** で自動的に適用されます。これにより、すべてのノードがエポック途中の不整合なしに状態遷移を準備できます。

```
エポック N     投票期間終了
エポック N+1   批准チェック実行
エポック N+2   発効: 新しい validator_id を BFT セットに追加
```

新しいバリデータの Falcon-512 公開鍵ハッシュ（`validator_id`）がオンチェーンのバリデータセットに追加され、エポック N+2 の最初のブロックからコンセンサスに参加します。

### 3.3 ガバナンストランザクション構造

```rust
struct GovernanceProposal {
    proposal_id          : [u8; 32],  // SHA3-256(tx_id || output_index)
    action_type          : u8,
    // 0x01 = バリデータ追加
    // 0x02 = バリデータ除去
    // 0x03 = パラメータ変更（ブロックサイズ・手数料フロア・エポック長など）
    // 0x04 = 憲法改正（アーカイブガバナンス承認が追加で必要）
    target_validator_id  : [u8; 32],  // 候補者の Falcon-512 公開鍵ハッシュ
    deposit              : u64,       // ロックされた MSK トークン量
    anchor_url           : Vec<u8>,   // オフチェーン提案理由文書の URL
    anchor_hash          : [u8; 32],  // 提案理由文書の SHA3-256 ハッシュ
    voting_epoch_start   : u64,       // 投票開始エポック
    voting_epoch_end     : u64,       // 投票終了エポック
}
```

`action_type 0x04`（憲法改正）は、通常のバリデータクォーラムに加えて **アーカイブガバナンスの 3/4 超多数決** を必要とし、単純多数による根本方針の書き換えを防止します。

---

### 3.4 投票の重み付けとデッドロック対策

**ステーク加重投票**: 各バリデータの投票は、`voting_epoch_start` 時点のスナップショットにおけるステーク MSK 残高に応じて重み付けされます。これにより経済的インセンティブとガバナンス結果が連動します。

```
effective_weight(バリデータ i) = staked_MSK_i / total_staked_MSK
weighted_yes = Σ effective_weight(i)  （vote_i == Yes の全 i）
```

批准条件: `weighted_yes ≥ 2/3`

**デッドロック対策**: `voting_epoch_end` までにクォーラムに達しなかった提案は **期限切れ** となり、デポジットが返却されます。提案者は修正したアンカーを添えて再提出できます。強制的なエスカレーションパスは意図的に設けていません。ネットワークの安定性を強制的な進行よりも優先するためです。

## 経済モデル（トークノミクス）

### 4.1 手数料構造

PQ 署名は大きなサイズを持ちます（Falcon-512: 約1.3 KB、LaRRS リング証明: 入力あたり約12〜24 KB）。手数料モデルはバイトコストを反映しつつ、プライバシーを高コストにしないよう設計されています。

**手数料計算式**

```
tx_fee = max(FEE_FLOOR, tx_size_bytes × FEE_PER_BYTE)

FEE_FLOOR    = 0.001 MSK   （サイズに関わらず最低手数料）
FEE_PER_BYTE = 0.000002 MSK/byte
```

100 KB の典型的なトランザクションの場合:

```
fee = max(0.001, 100_000 × 0.000002) = max(0.001, 0.200) = 0.200 MSK
```

`FEE_FLOOR` と `FEE_PER_BYTE` はいずれもガバナンスパラメータ（action_type `0x03`）であり、バリデータ投票によって変更できます。

**取引金額ベースの手数料**（価値ベーストランザクションへの適用）:

バイトベース手数料が低い場合、出力金額の **2%** を代替フロアとして適用します。

```
fee = max(バイトベース手数料, 0.02 × output_value)
```

この 2% の総手数料は以下のように3つの報酬プールに分配されます。

---

### 4.2 報酬分配

各ブロックで集めた手数料は `DistributedCoinbase` トランザクションを通じて再分配されます。

| 受取先 | 配分割合 | 根拠 |
|--------|---------|------|
| **プルーニングノード（バリデータ）** | 取引金額の 1.5%（手数料の 75%） | ブロック生成・BFT コンセンサス・リング署名検証 |
| **アーカイブノード** | 取引金額の 0.5%（手数料の 25%） | 完全履歴保存・ブートストラップ提供・エクスプローラー運用 |
| **管理者 / 財団** | ベース手数料のオーバーヘッド＋端数 | プロトコル保守・ガバナンスインフラ |

**バリデータ** プール内の配分は各バリデータの **ステーク MSK** に比例します。

```
バリデータ i の報酬 = pool_amount × (staked_MSK_i / total_staked_MSK)
```

ブロック提案者は自身のステーク比例分以外の追加ボーナスを受け取りません。これにより selfish mining のインセンティブを排除します。

新規トークン発行によるブロック報酬は存在しません。総供給量はジェネシス時に固定されており、流通する報酬は手数料のみで賄われます。

---

### 4.3 スラッシングポリシー

Misaka は **Cardano の没収なしペナルティモデル** を採用します。ステークの没収（slashing）は行いません。代わりに、不正行動をしたバリデータは **報酬減額とセットからの除去** という処置を受けます。

| 違反行為 | ペナルティ |
|---------|-----------|
| **二重署名（エクイボケーション）** | バリデータセットから即座に自動除去。当エポックの未払い報酬を没収 |
| **活動停止（連続100ラウンド以上不在）** | 離脱ラウンドの報酬倍率が 0 に設定。200ラウンド継続でガバナンスフラグが立つ |
| **無効なブロック提案** | 当ラウンドの報酬を差し止め。繰り返し発生でガバナンスによる除去投票をトリガー |

**没収しない理由**: バリデータ数が少ない（4〜16台）環境でのステーク没収は、バグやネットワーク分断による壊滅的な経済的損失リスクを生みます。報酬減額は同等の抑止効果をネットワーク崩壊リスクなしに実現します。

スラッシング記録はオンチェーンの `SlashRecord` として永続保存されます。トークン没収なしでも、評判上の説明責任は長期にわたって維持されます。

## アーカイブノードの信頼モデル

### 分散化要件

アーカイブノードはオペレーター管理（財団・取引所・パブリックグッド運営者）ですが、プロトコルは以下の要件を強制します。

- **最小アーカイブセット**: ガバナンスパラメータにより、メインネットでは独立したアーカイブオペレーターが ≥ 3 台必要
- **ブートストラップの多様性**: 新規ノードは ≥ 2 つの独立したアーカイブソースから UTXO スナップショットを取得し、一致を確認してから信頼します
- **検閲耐性**: アーカイブへの接続が 60秒以内に確立できない場合、バリデータからブロックヘッダーを同期するフォールバックに切り替わります（低速ですが検閲不可能）

### スナップショットの正当性検証

新規ノードは zstd 圧縮された UTXO スナップショットを受け取ります。全履歴のダウンロードなしに、以下の手順で正当性を検証します。

1. アーカイブノードが `{utxo_snapshot, snapshot_height, merkle_proof}` を提供
2. 新規ノードは ≥ 2 つのバリデータから `snapshot_height` のブロックヘッダーを独立して取得
3. `SHA3-256(utxo_snapshot)` を再計算し、`header.utxo_root` と一致することを確認
4. ブロックヘッダーの BFT 証明書（⌊2N/3⌋ + 1 以上の Falcon-512 バリデータ署名）を検証

BFT 証明書自体が信頼のアンカーとなるため、ZKP なしでも**暗号的にスナップショットと BFT 確定済みチェーン状態が紐付け**られます。将来的には UTXO セット導出の ZK 簡潔証明を追加し、アーカイブによる不正な代替チェーン提示への耐性をさらに強化する予定です。

## ポスト量子暗号パラメータ

### 6.1 Falcon-512

用途: バリデータブロック署名、BFT 投票、ガバナンス投票、ウォレット支払い認証

| パラメータ | 値 |
|-----------|-----|
| 格子次元 `n` | 512 |
| 法 `q` | 12289 |
| NTRU 格子 | あり（2の冪乗次数の円分環） |
| 署名サイズ | 約666バイト（圧縮）/ 1281バイト（パディング） |
| 公開鍵サイズ | 897バイト |
| セキュリティレベル | ポスト量子128ビット（NIST レベル I） |
| 署名ガウス幅 `σ` | ≈ 165 |
| ハッシュ to ポイント | SHAKE256 |

Falcon-512 は NIST PQC 標準化された署名スキームです（FIPS 206 ドラフト）。他の格子スキームと比較してコンパクトな署名サイズを誇り、Misaka のコンセンサス層における高頻度の BFT メッセージ送受信に適しています。

### 6.2 Kyber-768

用途: P2P Noise XX ハンドシェイク、ウォレットステルスアドレスの鍵カプセル化（Jamtis KEM ステップ）、メモ暗号化

| パラメータ | 値 |
|-----------|-----|
| モジュールランク `k` | 3 |
| 多項式次数 `n` | 256 |
| 法 `q` | 3329 |
| ノイズ分布 `η₁` / `η₂` | 2 / 2 |
| 暗号文サイズ | 1088バイト |
| 公開鍵サイズ | 1184バイト |
| 共有秘密サイズ | 32バイト |
| セキュリティレベル | 古典180ビット / ポスト量子164ビット（NIST レベル III） |

CRYSTALS-Kyber は NIST PQC KEM 標準です（FIPS 203）。レベル III は最低限要件の128ビットを大きく上回るセキュリティマージンを提供し、Module-LWE への将来的な暗号解析の進展を吸収します。

### 6.3 LaRRS リング署名パラメータ

LaRRS（Lattice Ring Signature Scheme）は Module-LWE 上に構築されたリング署名であり、Misaka のトランザクション匿名性層のためのポスト量子リング署名を提供します。

| パラメータ | 記号 | 値 | 備考 |
|-----------|------|-----|------|
| モジュールランク | `k` | 3 | Kyberと同じモジュール構造 |
| 多項式次数 | `n` | 256 | NTT フレンドリー |
| 法 | `q` | 8380417 | = 2²³ − 2¹³ + 1（素数、NTT フレンドリー） |
| L2 ノルム境界 | `β` | 2¹⁸ | 応答ベクトルのリジェクションサンプリング境界 |
| リジェクション率 | — | 約26%/ラウンド | ガウス分布の末端確率 |
| チャレンジ空間 | `c` | SHA3-256による256ビット | Fiat-Shamir 変換 |
| リングサイズ | `N` | 11〜16 UTXO | 最大匿名性のデフォルト: 16 |
| 証明サイズ (N=16) | — | 約11〜14 KB/入力 | リングメンバーごとの応答ベクトル |
| セキュリティレベル | — | ポスト量子128ビット | Module-LWE 困難性のもとで |
| セキュリティ前提 | — | MLWE、SIS | Module Short Integer Solution への帰着 |

**リンクタグの構成**

```
link_tag = SHA3-256(H_tag(sk_spend) || tx_nonce)
```

`H_tag` はドメイン分離されたハッシュ to 格子関数です。リンクタグは特定の支払い鍵に対して決定論的に生成され、トランザクションごとに一意となるため、鍵の同一性を明かすことなく二重使用を検出できます。

**証明サイズのトレードオフ**

リングサイズを 11 から 16 に増やすと、証明あたり約4 KB 増加しますが、匿名集合が 45% 拡大します。Misaka の 50〜200 KB のトランザクション予算内では、デフォルトとしてリングサイズ 16 を推奨します。

### 6.4 決定論的乱数生成（DRBG）

Misaka におけるすべての署名操作は、ポスト量子安全なソースからシードを取得した **Hash-DRBG（SP 800-90A Rev.1）** を使用します。

```
seed_material = OS エントロピー（256ビット、/dev/urandom またはプラットフォーム相当）
               || アプリケーションコンテキスト文字列
               || タイムスタンプノンス

DRBG = SHAKE256-DRBG(seed=seed_material, security_strength=256)
```

**なぜ CTR-DRBG（AES）ではなく SHAKE256-DRBG なのか**

CTR-DRBG は AES に依存しており、グローバー（Grover）アルゴリズムにより有効な鍵セキュリティが128ビットから64ビットに半減します。SHAKE256-DRBG には量子アルゴリズムが利用できる代数的構造がなく、そのセキュリティは SHA3 の衝突耐性に帰着します。グローバーによる影響は平方根係数の低下のみに留まり、256ビット出力幅においても ≥ 128ビットのポスト量子安全性が維持されます。

**決定論的鍵導出**（ウォレット）

```
master_seed  ← BIP-39 ニーモニック → エントロピー（256ビット）
spend_key    ← HKDF-SHA3-256(master_seed, info="misaka_spend_v1", len=64)
view_key     ← HKDF-SHA3-256(master_seed, info="misaka_view_v1",  len=64)
addr_key     ← HKDF-SHA3-256(view_key,    info="misaka_addr_v1",  len=32)
```

SHA3-256 を用いた HKDF により、導出された鍵が一つ漏洩してもマスターシードは明かされません（一方向性 HKDF 抽出）。また、すべてのウォレット鍵はニーモニック単体から再現可能です。

**エフェメラル署名ノンス**（Falcon-512）

Falcon のトラップドアサンプラーは内部でツリーベースのガウスサンプラー（NTRU-solve）を使用します。各署名操作の乱数は SHAKE256-DRBG インスタンスから新規に取得され、再利用されることはありません。Falcon におけるノンスの再利用は秘密鍵漏洩に直結するため、DRBG 状態は各署名呼び出しの前にアトミックに更新・永続化されます。

## トランザクションフォーマット

```rust
struct Transaction {
    version     : u8,
    inputs      : Vec<TxInput>,       // 各入力にリングとリンクタグを含む
    outputs     : Vec<TxOutput>,      // ステルスアドレス＋コミットメント
    ring_proof  : PQRingProof,        // LaRRS 証明

    link_tags   : Vec<[u8; 32]>,      // 二重使用防止

    tx_extra    : Vec<u8>,            // 暗号化メモ、エフェメラル KEM 暗号文
}

struct TxInput {
    ring        : Vec<[u8; 32]>,      // N 個の UTXO コミットメント（デコイ＋本物）
    link_tag    : [u8; 32],
    enote_image : [u8; 32],
}

struct TxOutput {
    stealth_addr: JamtisStealthAddr,
    commitment  : [u8; 32],           // 格子 Pedersen コミットメント
    enc_amount  : [u8; 48],           // ChaCha20-Poly1305 暗号化金額
    view_tag    : u8,                  // 1バイトのスキャン最適化ヒント
}
```

## ノードの起動

```bash
# クローンとインストール
git clone https://github.com/misaka-network/misaka-node
cd misaka-node
npm install

# ノード種別の設定
cp config/example.json config/node.json
# nodeType = "validator" | "light" | "archive" を設定
```

## セキュリティ前提

すべての暗号コンポーネントは以下のよく研究された困難問題に帰着します。

| プリミティブ | 困難問題 |
|------------|---------|
| Falcon-512 | NTRU / 2の冪乗次数円分環上の Ring-SIS |
| Kyber-768 | Module-LWE（MLWE） |
| LaRRS リング署名 | MLWE + Module-SIS |
| SHAKE256-DRBG | SHA3 の衝突耐性 |
| 格子 Pedersen コミットメント | SIS（Short Integer Solution） |

いかなるコンポーネントも離散対数問題や素因数分解に依存しておらず、Shor のアルゴリズムに対して完全な耐性を持ちます。
